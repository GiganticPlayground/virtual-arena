How the Virtual Arena prototype works

A walk-through of the runtime: how a webcam frame becomes a composited arena scene at 60 FPS while inference runs in a separate worker. Jump back to the prototype or the segmenter model comparison.

1. Overview 2. Threads & ownership 3. Preload 4. Per-frame pipeline 5. Render / infer decoupling 6. Compositor shader 7. Foam finger overlay

1What the prototype does

Capture a webcam (or uploaded video) frame, segment the person from the background, composite them onto an arena image, and optionally overlay a foam finger on each detected hand. All in the browser, all at interactive rates.

Inputs
Webcam via getUserMedia, bundled ./input_videos/*.mp4, or user-uploaded file.
Outputs
Composited canvas at display refresh rate + foam-finger DOM layer that tracks hand motion.
Keeps interactive
  • All inference in a dedicated worker
  • Render rate decoupled from inference rate
  • GPU composition via WebGL2, not per-pixel JS

2Threads & ownership

After boot, the main thread owns the DOM, the <video>, and a per-rAF frame capture. Everything else lives in the worker: WebGL2 context, MediaPipe tasks, ONNX sessions, textures. The canvas is transferred to the worker once via transferControlToOffscreen().

3Preload

Before anything runs, preload.js streams every asset with a progress bar, transfers the MediaPipe model buffers to the worker, and hands over the OffscreenCanvas. ONNX models are lazy-loaded on first selection (4 MB – 42 MB each).

4The per-frame pipeline

Every requestAnimationFrame tick, main posts a cam frame + params to the worker. The worker uploads to GPU, optionally runs inference, draws, and replies. Two independent busy gates (render, infer) let rendering keep flowing while slow models catch up.

5Render rate vs inference rate

The canvas refreshes at display rate, but inference runs on a separate budget. Missing a mask update just means the previous mask is reused — the person "lags" slightly behind their motion, but the scene never freezes.

Gates (worker-client.js)
  • renderBusy — cleared on 'rendered'. One draw in flight at a time.
  • inferBusy — cleared on 'inferenceDone'. One inference in flight at a time.
  • Independent: render keeps flowing while inference is still running.
Rate budget (main.js)
  • Inference Rate slider sets infRate (Hz).
  • Per tick: request infer only if (now - lastInfer) ≥ 1000/infRate.
  • Foam on → alternate seg / hands so each infer tick costs one model.

6The compositor shader

One fragment shader does the whole compositing pass. It samples the background with cover-fit UVs, samples the cam under the user's size/offset, runs a 5×5 kernel over the mask, and blends. Same program for every segmenter — a single uniform (u_maskMode) selects categorical vs continuous-alpha interpretation.

Why one shader?
Swapping models at runtime would otherwise mean a program rebuild. Keeping the fragment branches behind u_maskMode / u_maskInvert lets the compiler hoist the conditional out of the per-pixel work and makes model switches instant.
Edge controls (shader uniforms)
  • u_feather — 5×5 tap radius in mask-texel units
  • u_trim — shifts the smoothstep band or subtracts alpha bias
  • u_mirror — flips cam U only (mask flips with it)

7The foam finger overlay

Hands aren't composited by the shader — they're DOM images on top of the canvas. A pooled <img> per hand is positioned with a CSS transform computed from the MediaPipe landmarks, using the same scale + offset the shader uses for the source. Result: the foam finger tracks the subject perfectly even when the source is zoomed or shifted.

Why a DOM overlay, not GL?
The foam finger is a single PNG, rotated + scaled per hand. DOM layout handles that for free, and it sidesteps alpha-premultiply headaches with the compositor. The cost is one CSS transform per visible hand per frame — effectively zero.
Keeping hand and subject aligned
  • Landmarks are in video-space UV, where y=0 is the top of the video
  • Canvas UV flips that on the y-axis
  • Both paths apply the same camScale + camOffset
  • Mirror toggle negates x only
